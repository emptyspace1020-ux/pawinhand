(포인핸드)_AX_Deep_Research_프롬프트

현재 Input → Decision → Outcome 구조로 MVP 페이지가 뽑아져 있는 상태
- []는 프로젝트별로 수정

전제
1) 현재 Input → Decision → Outcome 구조로 3페이지(구현 페이지) MVP가 정의되어 있다.
2) 단, 리뉴얼 프로젝트이므로 기존 콘텐츠/기능을 삭제하지 않는다.
3) 대신 모든 정보/기능을 아래 5가지 중 하나로 “처리”한다:
   [유지(Primary) / 재배치(Relocate) / 접기(Collapse) / 조건부 노출(Conditional) / 비활성 노출(Disabled but visible)]

AX 정의(분석 기준)
- AX(AI Experience)는 화면 디자인이 아니라 “결정 구조 설계”이다.
- AX 구조 공식은 Input → Decision → Outcome 이다.
- MVP는 작은 기능이 아니라 “하나의 결정이 끝나는 구조”다.
- KPI는 감정이 아니라 “행동으로 끝나는 지표”다.
- 성공적인 설계는 기능 추가가 아니라 “질문 제거(결정의 마찰 제거)”다.

분석 대상

  ※ 실제 IA/메뉴/유입경로 기준으로 가정이 맞는지 먼저 검증하고 진행할 것.

1) KPI 정의(1차/2차)

1-1. 이 사이트의 1차 KPI를 “행동 기준”으로 정의하라.
- 예시(가정): “입양 후보를 결정하고 보호소/센터에 문의(전화/메시지/신청)까지 완료”
- 단, 포인핸드는 ‘결정의 끝’이 웹/앱/전화/외부 신청서로 분기될 수 있으므로, ‘완료’ 정의를 화면/행동으로 명확히 쪼개라.

1-2. 2차 KPI가 있다면 사용자군/상황별로 분리하여 정의하라.
- 예: 
  - [실종 동물 사용자] 제보 등록/공유/연락
  - [보호/목격 사용자] 발견 정보 등록/보호 중 신고
  - [후원 사용자] 정기/일시 후원 결제 완료
  - [보호소 관계자] 공고 노출/연락 유입/정보 업데이트

1-3. KPI 간 충돌이 발생하는 지점을 설명하라.
- 예시 충돌:
  - 입양 탐색 흐름(빠른 스크롤/필터/상세 진입) vs 실종/제보 콘텐츠의 강한 노출(배너/팝업)
  - 후원/캠페인 CTA의 과도한 상단 점유 vs “동물 찾기/문의”의 즉시성
  - 스토리/콘텐츠 소비 유도 vs ‘지금 공고 중’ 탐색의 단기 목표

------------------------------------------------------------
2) 사용자 여정(AX 구조) 분석

2-1. 현재 사이트의 사용자 여정을 Input → Decision → Outcome으로 단계별 정리하라.
- (권장) 최소 2개 여정을 분리:
  A) 입양 탐색/문의 여정
  B) 실종/목격/보호 제보 여정(있다면)

2-2. 각 단계에서 Decision이 “명확히 끝나는지” 평가하라.
- Decision의 종료 조건이 ‘사용자 머릿속’인지, ‘화면 상 행동 완료’인지 구분하라.
- “더 찾아볼까?”가 반복되는 구간은 Decision 미완료로 간주하라.

2-3. Outcome이 화면에서 인지 가능한 ‘끝점’으로 제공되는지 평가하라.
- 예: 문의/신청/연락/저장/공유/길찾기/후원완료 등
- Outcome이 외부(전화, 외부 사이트 신청서, 앱 이동)로 나가는 경우:
  - 나가기 직전의 “확인/요약/다음 행동 안내”가 제공되는지 평가하라.

3) 메인(Home)에서 “입양 준비/보호소 방문/연락” 정보를 어떻게 다뤄야 하는가

다음을 분석하라:

3-1. Home의 주 Decision은 무엇인가?
- 후보 예시:
  - “입양 가능한 동물을 탐색할 카테고리/조건을 선택한다”
  - “실종/목격/보호 중 목적을 선택한다”
  - “후원/캠페인 참여를 결정한다”
- 실제 트래픽/유입 의도(검색 키워드/랜딩 URL) 가정을 세우고, 그 가정 하에서 주 Decision을 확정하라.

3-2. “입양 준비/보호소 방문/연락 정보(절차·준비물·운영시간·위치·전화/메시지)”는
- 주 Decision을 방해하는가, 보조하는가?
- 어떤 사용자에게만 보조인가? (초보 입양자/재방문/모바일/지역기반 등)

3-3. 리뉴얼 조건상 이 정보를 빼지 않는다면,
아래 5가지 처리 방식 중 무엇이 최적인가?
[유지(Primary) / 재배치(Relocate) / 접기(Collapse) / 조건부 노출(Conditional) / 비활성 노출(Disabled but visible)]

3-4. Home에서 “즉시 행동이 필요한 사용자(예: 실종/목격/현장 보호)” 흐름을 다룰 경우,
주 KPI(입양 탐색/문의)를 방해하지 않는 분기 설계 중 무엇이 가장 적절한가?
- 버튼 / 배너 / 카드 / 플로팅(또는 FAB) / 상단 탭
- 선택 근거를 “결정 마찰 감소” 관점으로 설명하라.
  (예: 시야 점유, 주 흐름 방해, 되돌아오기 비용, 스크롤 깊이, 오작동 위험)

3-5. 웹 vs 모바일에서 노출 우선순위를 다르게 해야 한다면 권장안을 제시하라.
- 모바일: 즉시 행동(전화/길찾기/제보)의 접근성을 우선할지,
  탐색 효율(필터/리스트)의 연속성을 우선할지 기준을 세워라.
- 웹: 비교/검토(탭/필터/스크랩) 중심으로 갈지,
  앱 유도(딥링크/QR) 중심으로 갈지 분기 제안하라.

5) Adoption notice board 및 Adoption Notice Details에서
   “입양 준비/보호소 방문/연락(위치/운영시간/문의/신청)”의 구조 설계

5-1. ‘입양 KPI(문의/신청/연락) 이후’ “방문 준비”는 어떤 Outcome을 만드는가?
- 예시 Outcome 후보(가정):
  - 보호소 연락 완료(통화/메시지)
  - 신청서 작성 링크로 이동 완료
  - 방문에 필요한 정보 저장/공유 완료(주소/운영시간/주의사항/준비물)
  - 길찾기 실행 완료
- 포인핸드 맥락에서 가장 현실적인 ‘끝점’을 1~2개로 축약하라.

5-2. 모바일에서 이 기능을 제공할 때,
플로팅 버튼 / 하단 고정 CTA / 탭 중 어떤 방식이 가장 적절한가?
- “상세 읽기”를 방해하지 않으면서,
  “문의/길찾기”는 가장 짧은 경로로 제공하는 배치를 제안하라.

5-3. (리뉴얼 조건) 기존 기능이 많아 페이지가 비대해질 경우,
5가지 처리 방식(유지/이동/접기/조건부/비활성)으로
어떻게 정리할지 제안하라.
- 단, 삭제 금지. “결정에 필요한 정보”를 우선 식별한 후 처리하라.

6) KPI 비기여 요소 “제거”가 아니라 “처리”로 분류

중요: 삭제하지 않는다.

6-1. KPI 기여도가 낮은 요소를 찾아서,
어떤 처리 방식으로 정리할지(5가지 중 선택) 추천하라.
[유지(Primary) / 재배치(Relocate) / 접기(Collapse) / 조건부 노출(Conditional) / 비활성 노출(Disabled but visible)]

6-2. 각 추천에 대해 “왜”를 Decision 마찰 감소 관점에서 설명하라.
- 사용자가 ‘지금 결정을 끝내는 데’ 필요하지 않은 질문/선택지를 줄이는가?
- 탐색→상세→문의/신청 흐름이 더 짧아지는가?
- 주의: 사회적 가치/브랜딩 요소가 필요할 수 있으므로,
  “유지하되” 어떻게 마찰을 줄일지(접기/이동/조건부)를 우선 검토하라.

7) 비교 사례 분석(벤치마크)

“문화기관/티켓 플랫폼” 대신,
유사 목적의 플랫폼/기관/서비스 5~8곳을 비교 분석하여 표로 정리하라:
- 유기동물 입양 공고/보호소 정보 제공
- 실종/목격/보호 제보
- 기부/후원/캠페인(선택)
- 콘텐츠/스토리(선택)

표에 포함할 비교 항목:
- 메인에서 ‘입양 탐색’과 ‘실종/제보’를 어떻게 다루는가(우선순위/분기 방식)
- 상세 페이지 CTA 위치/형태(특히 모바일: 하단 고정/플로팅/상단/탭)
- 탐색→상세→문의/신청 흐름(단계 수/마찰)
- “문의/신청 이후” 방문 준비 정보의 위치(상세 내/별도 안내/마이페이지/팝업/외부링크 등)
- 출처 링크 포함(각 서비스의 해당 화면/가이드 페이지 링크)

※ 벤치마크 선정 시:
- 국가/공공(공고 제공) 1~2
- 민간 플랫폼 2~3
- NGO/단체(입양/후원) 1~2
- (가능하면) 앱/웹 모두 참고

8) 최종 산출물 형식(반드시 이대로)

1. KPI 정의(1차/2차)
2. AX 구조 표(Input/Decision/Outcome)
3. Decision 모호 구간 & 해결 방향
4. 3페이지 IA 제안(Home / Adoption notice board / Adoption Notice Details)
5. 콘텐츠/기능 처리표(유지/이동/접기/조건부/비활성)
6. 웹/모바일 노출 우선순위 권장안
7. 벤치마크 표 + 출처
